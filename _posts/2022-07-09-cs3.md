# 디버깅
* 코드 작성 후 오류발생 시 오류와 원인을 확인하는 것
* 디버깅은 배포와 반대의 개념으로 개발중인 상태를 나타내는 단어이기도 함

<br>

# 컴파일러 vs 인터프리터
사람이 읽을 수 있는 언어로 코드 작성 후 기계가 알아들을 수 있도록 번역해줘야함
  
## 컴파일러
* 작성한 코드들을 배포 전 기계어 혹은 중간언어로 번역하는 것
* 빌드 과정중에 포함되어 있음
* 대표적으로 java로 작성 후 JVM용 언어로 번역

## 인터프리터
* 작성한 코드들을 실행할 때마다 번역하는 것
* 컴파일할 필요가 없기 떄문에 코드 수정이 잦은 서버의 경우 업데이트가 수월하다는 장점
* 단, 실행할 때마다 번역하기 때문에 컴파일러 언어에 비해 느리다는 단점
* 프로그래머가 짠 코드가 그대로 배포되기 때문에 코드가 노출됨
* javascript,python,ruby (= 스크립트 언어)

<br>

# 빌드
작업하는 프로젝트 파일들을 출시하기 적합한 형태로 포장하는 일

## 목적과 이유
1. .exe 파일(실행 파일)로 제공하기 위함일 때
2. 코드 미니멀화
3. 난독화 

<br>

# 배포
코딩해서 빌드한 결과물을 사용자에게 전달하는 과정

* 데스크탑 프로그램이라면 다운받을 수 있는곳에 업로드
* 모바일앱이라면 플레이 스토어나 앱스토어에 업로드
* 웹사이트 경우에는 서버에 올린다음 실행해서 사람들이 해당 도메인으로 접근해서 볼 수 있도록 하는 작업
  * 웹개발에서 배포란 사람들이 보는 웹사이트에 작업한 바를 반영한다는 의미

<br>

# 환경 변수
컴퓨터마다 **지정된 값**
## 환경
* 소프트웨어가 동작하는 공간 (소프트웨어 = 컴퓨터)
* 도커의 컨테이너 등 프로세스가 돌아가는 세분화된 공간

## 환경변수의 용도
1. 명령어 입력 시 환경변수 값을 찾아서 각자 원하는 값이 나오게함
2. 환경의 조건마다 어떤 동작을 시행할 지, 어떻게 시행할 지를 정할 수 있음 (작업시 개발용, 배포용의 환경변수를 다르게 적어서 용도에 맞게 작업)
3. 오픈소스로 공개되는 작업물인 경우 공개되면 안되는 파일들을 따로 환경변수로 지정해서 보안 유지를 할 수 있음

<br>

# 라이브러리 vs 프레임워크
## 라이브러리
* 어떤 소프트웨어든 필요시 **가져다** 붙여넣을 수 있도록 모듈화된 코드, 또는 그 묶음
* 나만의 라이브러리를 만들 수 있음
* 건축으로 치면 건축의 자재들

## 프레임워크
* 소트프웨어의 기초 설계가 프로그래밍 되어 있고 이를 다운받아 **규칙에** 맞게 그 위에 덧붙여서 내가 원하는 프로그램을 만듦
* 기본 골격을 갖춘 집 조립 키트

  ### 프레임워크 종류
  * 스프링
  * 장고
  * .NET
  * 라라벨
  * Express
  * Vue

## 경계가 애매한 것
* React : 프레임워크처럼 보이지만 필요할 때 가져다 붙일 수 있는 라이브러리임
